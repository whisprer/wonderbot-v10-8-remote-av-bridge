Yep — the fallback line tells us the exact fault:

Your HF voice is failing because SpeechT5 is trying to load the Matthijs/cmu-arctic-xvectors speaker-embedding dataset, and newer datasets releases no longer support dataset loading scripts. The cmu-arctic-xvectors repo still relies on that old script path, so the loader blows up and WonderBot falls back to pyttsx3.

So you’ve got two viable fixes.

Fastest fix for the current repo
Pin datasets back to a pre-4.0 release so SpeechT5 can load the speaker embeddings again:

py -3.11 -m pip install "datasets==3.6.0"

Then relaunch WonderBot and check /diagnostics again. The reason this works is that the break happened in datasets 4.0.0+, where script-based dataset loading was removed.

Cleaner long-term fix
Stop using SpeechT5’s CMU-ARCTIC embedding path and switch the default HF voice model to facebook/mms-tts-eng instead. MMS-TTS is supported in Transformers as a VITS-style TTS model and does not depend on the CMU-ARCTIC xvector dataset path that is breaking you.

The trade-off is important:

SpeechT5 is the better licensing default here; its model card is MIT-licensed, but its common speaker-embedding workflow is what’s breaking.
MMS English is easier to run cleanly in current Transformers, but its model card is CC-BY-NC-4.0, so it is a bad default if you want commercial freedom later.

So my recommendation is:

If you want the quickest recovery right now: pin datasets==3.6.0.
If you want the least annoying fully local free path and non-commercial is acceptable: switch to facebook/mms-tts-eng.
If you want to preserve commercial-friendliness: keep SpeechT5 and pin datasets.
